# vnpy_qmt 源码 Vendor 接入设计

日期：2026-04-17

## 背景

当前项目已经完成了以 `xtquant -> vnpy_xt -> stock_a -> AlphaLab -> BacktestingEngine` 为主线的历史数据验证，下一阶段需要逐步补齐 QMT 实盘接入能力。

在调研过程中，`vnpy_qmt` 被识别为一个第三方的 vn.py QMT 网关项目，其价值主要在于：

- 提供基于 vn.py `BaseGateway` 风格的 QMT 交易接入参考；
- 提供 `connect`、`subscribe`、`send_order`、`cancel_order`、`query_account`、`query_position`、`query_order`、`query_trade` 等交易网关侧接口映射；
- 可以作为后续“优先复用现有实现，而不是从零写 xttrader 网关”的基础。

用户当前的诉求不是立刻改造实盘流程，而是先把 `vnpy_qmt` 源码拉到项目内，便于：

- 本地统一管理；
- 结合当前项目做定制调整；
- 后续明确地跟踪上游更新。

## 目标

本次交付目标是把 `vnpy_qmt` 以“可维护源码快照”的方式纳入当前 `xt-first` 工作树，而不是直接完成安装接入或实盘联调。

需要完成：

1. 将 `vnpy_qmt` 上游源码放入当前工作树的 `vendor/vnpy_qmt`；
2. 保留上游原始项目结构，避免这一步就进行目录重构；
3. 在源码目录中记录上游仓库地址、拉取时间、基准 commit 或 tag、后续更新方式；
4. 让后续开发可以在项目内直接阅读、修改、比对这份源码；
5. 不影响当前已经跑通的 `vnpy_xt` 历史数据链路。

## 方案选择

### 方案 A：纯源码快照

直接把上游仓库完整复制到项目内。

优点：

- 最直接；
- 落地最快。

缺点：

- 后续不容易追踪来源；
- 容易忘记当前快照对应哪一版上游代码。

### 方案 B：源码快照 + 上游元信息记录

将上游源码放到 `vendor/vnpy_qmt`，同时新增 `UPSTREAM.md`，记录来源、版本基线和同步方式。

优点：

- 仍然足够简单；
- 后续更容易做本地修改和上游比对；
- 不引入 submodule 的额外复杂度。

缺点：

- 后续上游同步仍然需要人工执行。

### 方案 C：git submodule

把上游仓库作为 submodule 引入。

优点：

- 上游版本关系更标准；
- 同步历史更清楚。

缺点：

- 会增加主仓库的管理复杂度；
- 对当前“先快速纳入并改造”的目标不够顺手。

本次选择方案 B。

## 目录设计

本次固定采用以下目录：

- vendor 目录：`C:\Users\sai\project\vnpy\.worktrees\xt-first\vendor\vnpy_qmt`
- 上游说明文件：`C:\Users\sai\project\vnpy\.worktrees\xt-first\vendor\vnpy_qmt\UPSTREAM.md`

设计原则：

- 放在当前 `xt-first` 工作树内，而不是主仓库根目录；
- 与当前 XT/QMT 研发线保持隔离；
- 不污染其它尚未切换到这条路线的工作区；
- 后续如果方案成熟，再决定是否上移到主仓库公共目录。

## 内容边界

### 本次要做的事

1. 拉取 `vnpy_qmt` 上游源码；
2. 将源码放入 `vendor/vnpy_qmt`；
3. 写入 `UPSTREAM.md`；
4. 检查目录结构是否完整可读。

### 本次不做的事

1. 不修改当前主项目的安装脚本；
2. 不把 `vendor/vnpy_qmt` 立即接入 `pyproject.toml`；
3. 不自动安装 `vnpy_qmt` 到当前 Python 环境；
4. 不在这一步直接打通实盘交易；
5. 不重构 `vnpy_qmt` 内部代码风格或包结构。

## 上游记录设计

`UPSTREAM.md` 至少应包含以下信息：

- 上游仓库地址；
- 拉取日期；
- 本地纳入时使用的 commit hash 或 tag；
- 当前纳入方式为 vendor 快照，而非 submodule；
- 后续更新建议流程。

建议的后续更新流程：

1. 在临时目录重新拉取上游最新代码；
2. 对比当前 `vendor/vnpy_qmt` 与上游差异；
3. 先保留本地定制修改，再人工合并上游更新；
4. 更新 `UPSTREAM.md` 中的 commit 记录。

## 风险与约束

### 1. 上游项目不是官方 vn.py 插件

含义：

- 后续兼容性需要我们自己验证；
- 不能假定它和当前主仓 `vnpy`、`vnpy_xt`、`xtquant` 完全无缝匹配。

### 2. vendor 后容易与“已安装包”混淆

含义：

- 把源码放进项目，不等于当前环境已经在使用它；
- 后续需要明确“阅读参考”和“实际运行接入”的边界。

### 3. 后续本地改动会偏离上游

含义：

- 必须保留上游来源记录；
- 更新时要能区分“上游原始变化”和“本地定制变化”。

## 验收标准

本次完成后，应满足：

1. `vendor/vnpy_qmt` 在当前工作树存在；
2. 目录中包含上游源码主文件；
3. `UPSTREAM.md` 记录完整；
4. 当前项目其它运行链路未被修改；
5. 可以基于这份源码继续做后续兼容性分析与定制开发。

## 结论

当前最合适的做法，是先把 `vnpy_qmt` 作为一个带来源记录的 vendor 源码快照放进：

`C:\Users\sai\project\vnpy\.worktrees\xt-first\vendor\vnpy_qmt`

这样既能满足“放在项目内方便管理和调整更新”的目标，又不会在当前阶段把安装方式、依赖管理和实盘接线问题混在同一步里。
